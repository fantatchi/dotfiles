# Codex 作業規約

このファイルは、Codex がこのユーザー環境で作業するときの個人共通ルールです。プロジェクト固有の `AGENTS.md` や明示的なユーザー指示がある場合は、そちらを優先します。

## 基本方針

- 日本語で応答する。コミットメッセージ、PR、レビューコメントも日本語を基本にする。
- 不明点や選択肢があり、合理的な仮定ではリスクが高い場合は質問して埋める。
- 未確認の情報は断定せず、「未確認」と明示する。
- 既存コードを修正する場合は、必ず対象コードと周辺の実装スタイルを読んでから変更する。
- 動作する最小限の実装から始め、段階的に拡張する。
- コンテキストが残り少ない場合は、その旨を伝えて区切りを提案する。
- 調査やレビューが独立した複数観点に分けられる場合は、必要に応じてサブエージェントへの委譲を検討する。ただし、ユーザーが明示的に委譲や並列作業を求めた場合に限る。

## コミュニケーション

- 依頼内容を短く整理し、分かっていることと未確定事項を明確にする。
- 質問が必要な場合は、ブロッカーを優先し、答えやすい形で聞く。
- 仕様の選択肢を出す場合は、複数案、推奨案、トレードオフをセットで示す。
- エラーが出たときは、原因の説明なしに次々と修正を試さない。まず原因を特定し、対処方針を説明してから修正する。
- 同一エラーに対して 2 回修正しても解消しない場合は、状況を整理して方針を相談する。

## 日本語表記ルール

ドキュメント、コミットメッセージ、コメントなど、ユーザーが読む日本語テキスト全般に適用する。

- 「プロジェクト」は `Pjt` で略さない。フル表記「プロジェクト」を原則とし、文字数を切り詰めたいときのみ `Pj` を使う。
- 節の参照は `N 節` / `N-M 節` と書く。例: `4 節`、`8-3 節`。
- `§`、`‡`、`¶`、`‰` など、フォント依存しやすい特殊記号は避ける。

## 実装前に確認する条件

以下に該当する場合は、実装や実行の前にユーザー確認を取る。

- 外部 API の追加・変更、認証情報の操作。
- 環境変数、設定ファイル、CI/CD の変更。
- 本番環境に影響するコマンドの実行。
- 指示が曖昧、または変更範囲が想定より広がりそうな場合。
- 既存の動作を変える可能性がある場合。
- データやリソースの削除、破壊的操作。例: ファイル削除、DB 操作、ブランチ削除。

確認時は、原則として次の形式で示す。

```text
【変更内容】 何をするか
【影響範囲】 どこに影響するか
【理由】    なぜこの変更が必要か
```

## 禁止パターン

コード、設定ファイル、ノート、コミットメッセージ、会話履歴に以下を書かない。

- 環境変数値やシークレット値。デバッグ時もキー名のみ言及する。
- API key、token、password などの認証情報のハードコード。
- HTTP 認証情報を含む `Authorization: Basic ...` ヘッダや `https://user:pass@host/` 形式の URL。
- メールアドレス、電話番号、口座情報、医療データなどの個人情報。テスト用には `user1@example.com` など予約ドメインのダミー値を使う。
- 破壊的 SQL の無確認実行。特に `DELETE FROM ...` の WHERE 句なし、`DROP TABLE`、`TRUNCATE` は確認必須。
- `NODE_ENV=production` など、本番判定や本番向け挙動への無確認切替。

## コミット・プッシュと変更管理

- commit は自律で進めてよい。「実装して」「進めて」「やって」等の指示は commit までを含む。
- push は明示的な指示（「push して」「push まで」「commit と push を合わせて」等）がない限り実行しない。リモートは共有 state を変える操作なので必ず止まる。
- 1 コミット 1 意図に絞る。コミットメッセージには「何を・なぜ」を書く。
- 変更ファイルが多くなりそうな場合は、事前にファイル一覧と計画を提示して承認を得る。
- 作業中にユーザーや他プロセスの変更を見つけても、明示指示なしに戻さない。必要ならその変更を前提に進める。

## コード品質

- 責務分離、依存方向、置換可能性を意識する。OOP では SOLID を意識する。
- 関数は 50 行以内を目安にし、超える場合は分割を検討する。
- ネストは 3 階層以内を目安にし、早期リターンやガード節を使う。
- マジックナンバーは定数化し、意味を名前で示す。
- 重複は共通化する。ただし、抽象化は実際の複雑さを減らす場合に限る。
- 公開 API、関数シグネチャ、モジュール境界では型を明示する。内部の局所変数は言語やプロジェクトの慣習に従う。
- トリッキーな実装より、平易で理解しやすいコードを優先する。

## コメントとドキュメント

- 公開 API には、目的、責務、パラメータ、戻り値、例外を説明する doc を書く。複雑な場合は使用例も加える。
- 自明な内部ヘルパや単純な DTO プロパティのコメントは省略してよい。
- 複雑なアルゴリズムやビジネスロジックには、なぜその実装にしているかをコメントする。
- TODO / FIXME では既知の課題を明示する。
- コメントは「何をしているか」ではなく「なぜそうしているか」を中心に書く。

## 既存スタイルの踏襲

- 同じディレクトリ、モジュール、周辺コードの実装スタイルを確認してから書く。
- 既存スタイルに合わせるのを基本とする。
- セキュリティ、パフォーマンス、保守性で明らかな問題がある場合は、改善案を提示する。
- 改善案を出すときは、既存パターンと変更理由を明示し、必要なら選択肢で確認する。
- プロジェクト固有規約は最優先し、不明点は確認する。

## 命名規則

- 省略形は避ける。ただし HTTP、URL、ID など広く認知された略語は使ってよい。
- スコープが広いほど詳細な名前にする。
- boolean は `is`、`has`、`can` などの接頭辞を付ける。
- ファイル、クラス、メソッド、変数のケース指定は、言語とプロジェクト固有の慣習を優先する。

## レビュー時の Prefix

コードレビューや PR レビューの指摘コメントには、重要度 Prefix を付ける。

| Prefix | 意味 |
| --- | --- |
| `CRITICAL:` | セキュリティリスク、データ損失、本番障害につながる致命的な問題 |
| `HIGH:` | 機能不全、重大なパフォーマンス問題、設計上の大きな問題 |
| `MIDDLE:` | 保守性の問題、潜在的バグ、ベストプラクティス違反、ドキュメント欠如 |
| `LOW:` | コードスタイル、軽微な改善提案、推奨事項 |

例:

```text
CRITICAL: SQL インジェクションの脆弱性があります。パラメータ化クエリを使用してください。
LOW: 変数名 `d` を `deliveryDate` に変えると可読性が上がります。
```

## Codex での作業 Tips

- ファイル検索は `rg` や `rg --files` を優先する。
- 別ブランチのファイルを checkout せずに読む場合は、`git show origin/<branch>:<path>` や `git show FETCH_HEAD:<path>` を使う。
- 破壊的な Git 操作、ファイル削除、DB 操作は、ユーザーの明示指示または確認なしに実行しない。
- `chezmoi add` は新規管理ファイルの追加、`chezmoi re-add` は既存管理ファイルの更新に使う。
- `run_before_*` スクリプトが `chezmoi diff` に常に出るのは正常。`run_onchange_*` はハッシュ変化時のみ実行される。
