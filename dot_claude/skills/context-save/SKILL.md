# コンテキスト保存

現在のプロジェクトの作業状態を保存し、次回セッションで復帰できるようにする。

このスキルは **コア（`.claude/context.md` の保存だけで完結・外部依存なし）** と **連携（tasks.md / progress.md / MEMORY などが揃っていれば実行する任意処理）** に分かれる。Obsidian や他スキルが無い環境でも「## コア」だけで動作する。連携の配線は `~/.claude/skills/shared/integrations.md`（resolver）で判定する。

## コア（単独完結・連携なしで動く）

### 書き出し先

プロジェクトルートの `.claude/context.md`（上書き保存）

#### プロジェクトルートの決定

1. `git rev-parse --show-toplevel` でリポジトリルートを取得
2. git リポジトリ外の場合は CWD をプロジェクトルートとする

#### ディレクトリの確認

- `.claude/` ディレクトリが存在しなければ `mkdir -p` で作成する

### 保存内容

`./template.md` のフォーマットに従って書き出すこと。

#### 情報の収集

以下の情報を収集してテンプレートに埋める：

- **project**: リポジトリのディレクトリ名（`basename` で取得）
- **git リモート URL**: `git remote get-url origin`（取得できない場合は空欄）
- **現在のブランチ**: `git branch --show-current`
- **git 状態**: `git status --short` の要約
- **直近のコミット**: `git log --oneline -5`
- **プロジェクト概要**: セッション中の作業内容から簡潔にまとめる
- **進行中の作業**: 現在取り組んでいるタスクや未完了の作業。**各 entry は `- YYYY-MM-DD（任意の補足）: 内容` 形式で書く**（日付プレフィックスがローテーションの基準。古い entry は「## 進行中の作業のローテーション」セクションで自動削除）。既存ファイルの entry がこの形式になっていない場合は、保存時に Claude が当時の日付を推定して付け直す（git log の日付・本文中の日付・前後の文脈から推定。推定できなければ「YYYY-MM-DD不明」のままにして保守的に残す）
- **判断メモ**: セッション中に行った重要な判断とその理由（プロジェクト固有の判断は context.md に残す。プロジェクトをまたいで再利用される普遍知見の MEMORY.md 昇格は「## 連携」の連携3 で扱う）
- **重要ファイル**: 作業に関連する主要ファイル（プロジェクトルートからの相対パス）
- **関連リポジトリ履歴**（ホームワークスペース用・任意）: `chezmoi source-path` でパスを取得し、`git -C <path> log --oneline -n 5` と `git -C <path> remote get-url origin` を「関連リポジトリ」セクションに埋め込む。コマンド失敗時（chezmoi が無い等）はスキップ

**次のステップについて**: 次回セッションで着手すべきタスクは context.md には書かない。連携が有効ならタスクストアへ吸い上げる（「## 連携」の連携1 参照）。連携が無効な環境では、必要に応じて context.md の `## メモ` 等に短く残す運用に委ねる。

### 既存ファイルの扱い

書き出し先に既存ファイルがある場合：
- まず既存ファイルを読み込む
- `## プロジェクト概要` は既存の内容を引き継ぎつつ、必要なら更新
- それ以外のセクション（ブランチ・状態、進行中の作業など）は最新情報で上書き
- `updated` の日時を更新する
- 過去に `## 次のステップ` セクションが残っている場合は**削除**する（タスクは連携先に一本化、context.md には書かない）

### 進行中の作業のローテーション

`## 進行中の作業` セクションは時系列で追記されるため放置すると肥大化し、context-save / context-load の token 消費・処理時間が増え続ける。保存時に古い entry を機械的に削除して定常サイズに保つ。

#### ルール

- **削除対象**: entry 先頭の日付（`- YYYY-MM-DD...` プレフィックス）が **今日から 14 日より古い** もの
- **最低保証**: 削除後の entry 数が 3 件未満になる場合、新しい順に 3 件は必ず残す（時系列の連続性を保つための保険）
- **情報源について**: 削除した entry の本文を残したい場合は、外部の作業ログ（別スキルで記録したログ・git log・手動メモ等）を併用する想定。本スキルは外部ログの存在を前提としない（context.md からの削除のみ責務）。外部ログを併用していない環境では、削除前に内容を確認するか、運用に合わせてローテーション閾値を伸ばす判断をユーザーに委ねる

#### 手順

1. 既存 context.md の `## 進行中の作業` セクションを抽出
2. トップレベルの `- YYYY-MM-DD` で始まる行を entry の境界として分割する（子の箇条書きやインデントされた行は直前の entry の一部として保持）
3. 各 entry 先頭の日付（`- YYYY-MM-DD` 部分のみ、`（午後・完了）` 等は無視）をパースし、今日との日数差を計算
4. 「14 日以内 (= 今日 − 14 日 以降)」の entry に加え、それでも 3 件未満なら新しい順に 3 件まで補充
5. 残った entry を新しい順で並べ、書き戻す
6. 削除した entry がある場合は、書き戻し後に「（古い N 件はローテーションで削除）」程度の 1 行注記を末尾に追記してもよい（任意）

#### 注意

- `## 判断メモ` セクションは時間でローテーションしない（再利用される普遍知見のため）。MEMORY.md への昇格判断は「## 連携」の連携3 を参照
- `## 関連リポジトリ` の「直近のコミット」リストは保存ごとに最新 5-6 件に丸める（既存挙動）
- 日付パースが失敗した entry（プレフィックスが想定形式と異なる）は **保守的に残す**（誤削除より誤残存を選ぶ）

### 判断メモの肥大アラート

`## 判断メモ` セクションは時間でローテーションしない（普遍知見の昇格判断が要るため機械削除できない）。代わりに、保存時にエントリ件数をチェックし、肥大したら圧縮を促すアラートを出す。

#### ルール

- `## 判断メモ` のトップレベル箇条書き（`- ` で始まる行。子のインデント箇条書きは数えない）を数える
- 件数が **15 件を超えている** 場合、context-save 完了後に次の 1 行アラートを表示する:
  - `⚠️ 判断メモが N 件あります。昇格・統合・削除での圧縮を検討してください`
- **アラートは常に 1 行**。連携4 が発火する場合は、この汎用 1 行を出さず連携4 の文言に**置換**する（2 行出さない）
- アラートを出すだけで、本スキルは判断メモを削除・編集しない（仕分けは判断を要するため別途。委譲先の案内は連携4）

## 連携（任意・対象があれば実行）

**連携の前提確認**: `~/.claude/skills/shared/integrations.md` を Read する（無ければ全キー未設定とみなす）。各連携の ［参照キー］ を resolver の「参照規約」で判定する。**(a)→(b)→(c) を上から評価し最初に真の分岐を採る**:
- **path 系キー**（task_store 等）: (a) 空/未設定 → skip / (b) パスあり＋`*_probe`（無ければキー自身）存在 → 実行 / (c) パスあるが probe 不在 → skip
- **bool 系キー**（progress_map / memory_promotion）: probe 判定なし。`on` → 実行 / `off`・未設定 → skip

いずれの連携も skip 可能で、「## コア」の成否には影響しない。

### 連携1: タスクストアへの Next 吸い上げ ［参照キー: task_store, task_store_probe］

セッション中に発生した「次回セッションで着手すべきアクション」を `task_store`（タスクストア tasks.md）の `## Next` セクションに追記する。context.md は project-local、タスクストアは cross-project の共有ストアであり、その橋渡しが context-save の責務。

**フォーマット規約は `~/.claude/skills/shared/tasks-format.md` を SSOT として参照する**。タイトル文字数規則（60-100 文字中心、150 文字絶対上限）・タスク行構造・プロジェクトタグ規約はすべて shared 側に従う（本 SKILL.md で再掲しない）。`gtd-add` は同じ規約に従って Inbox へ書き、本スキルは Next へ書く、という書き込み先の違いだけの分業。

#### 手順

1. resolver の `task_store` が指すファイルを Read で読む
   - `task_store` が空 → 連携1 を skip（context.md 本体の保存・ローテーションには影響しない）
   - `task_store_probe`（既定 `~/ObsidianVault/.obsidian/`）が不在 → 未同期とみなし skip
   - パスは指すがファイル自体が不存在 → 初期テンプレ生成は本スキルでは行わない（責務外。`/gtd-add` 等で初期化してから再実行を案内）
2. 現在プロジェクトのタグを決定: `#project/<basename of git toplevel>`（git 外なら CWD 名、`$HOME` 完全一致なら `#project/global`）
3. セッション中に発生した「次に着手すべきアクション」を抽出
4. **【文字数ガード｜MUST・書き込み前に必ず実行】** 追記しようとする各 Next 行について、**タイトル本体**（`- [ ] #project/<name> ` プレフィックスを除いた残り全体）の文字数を**実際に数える**。tasks-format.md の規約（60-100 文字中心・**150 文字絶対上限**）に照らし、**150 文字を 1 文字でも超える行はそのまま書き込まない**。超過時は次のいずれかで必ず短縮してから書く:
   - 詳細（手順・根拠・数値・commit ID・ファイル一覧・経緯）を `context.md` / `docs/` / `progress.md` へ逃がし、Next 行には**短い見出し + 参照ポインタ**だけ残す（例: `... (詳細 context.md 6/6)`）
   - **禁止アンチパターン**: context.md の段落・箇条書きをそのまま Next 行に貼り付けない。タスクストアは**タスク管理用**であり進捗ログ・判断メモを書く場所ではない（肥大化の主因。過去に 1,500 文字超の行が混入した実例あり）
   - 短縮しても 150 文字に収まらない場合は、タスクを複数行に分割する（1 タスク = 1 行 = 1 アクション）
5. 既に同内容のタスクが `## Inbox` / `## Next` / `## Waiting` にある場合は重複追加しない（部分一致で判定）
6. `## Next` セクションに `- [ ] #project/<name> タイトル` 形式で追記し、Edit で書き戻す
7. **【書き込み後の自己検証｜MUST】** 書き戻した直後に、追記した各 Next 行のタイトル本体文字数を再確認する。150 文字超の行が 1 つでも残っていたら、その場で短縮して再度 Edit する（検証なしで完了としない）

#### 追加対象の判断基準

- 明確に「次にやること」として合意されたアクションのみ追加する
- 推測や「やったほうがいいかも」レベルは追加しない（思いつき・未分類は Inbox 起きの判断として `gtd-add` 側に委ねる）
- タスクが発生していない場合はスキップしてよい

### 連携2: 進捗マップの更新 ［参照キー: progress_map］

`progress_map` が on で、プロジェクトルートの `.claude/progress.md` が存在する場合、セッション中の作業を反映してチェックボックスを更新する。`.claude/progress.md` はローカル管理（`.gitignore` 対象）で、各開発者が個別に保持する。

#### 手順

1. `{project-root}/.claude/progress.md` を Read で読む
   - `progress_map` が off → 連携2 を skip
   - 存在せず、かつ `{project-root}/CLAUDE.md` に `## 進捗マップ` セクションがある場合は、後方互換として CLAUDE.md 側を対象にする（旧形式）
   - どちらもない場合はスキップ
2. セッション中に完了した作業に対応するチェックボックスを `- [ ]` → `- [x]` に更新する
3. 新たに着手が見えたタスクがあり、進捗マップに項目がない場合は適切な位置に `- [ ]` で追加する
4. `最終更新:` の日付を当日に更新する
5. Edit ツールで書き戻す

#### ルール

- 進捗マップの構造（セクション番号・見出し）は変更しない
- チェックの付け外しと項目の追加のみ行う
- 既にチェック済みの項目を外さない
- 不確かな場合はチェックしない（次回セッションで確認）

### 連携3: 判断メモの MEMORY.md 昇格提案 ［参照キー: memory_promotion］

`memory_promotion` が on の場合、`## 判断メモ` のうちプロジェクトをまたいで再利用される判断（コミット粒度ルール、ツール固有の罠、ライブラリ選定基準など普遍的知見）について、MEMORY.md（`feedback_*.md` / `user_*.md` / `reference_*.md`）への昇格を提案する。プロジェクト固有の判断は context.md に残す。判断メモは時間でローテーションしないため、context.md に書き続けると肥大化の主要因になる点を踏まえて仕分けを促す。

- `memory_promotion` が off / 未設定 → 連携3 を skip（昇格提案を出さない）
- 昇格の実書き込みは本スキルでは行わず、提案のみ（ユーザー判断 or MEMORY 管理フローに委ねる）

### 連携4: 判断メモ圧縮の委譲案内 ［参照キー: session-review スキルの有無］

コアの「判断メモの肥大アラート」が発火した（15 件超）とき、`session-review` スキルが利用可能なら、コアの汎用 1 行を次の文言に**置換**する（追記しない＝アラートは常に 1 行）:

- `⚠️ 判断メモが N 件あります。/session-review の「判断メモ圧縮」フェーズで昇格・統合・削除を検討してください`

`session-review` が無い環境では、コアの汎用アラート文言のままにする（委譲先を案内しない）。

## 注意事項

- パスはプロジェクトルートからの相対パスで記録する（マシン非依存）
- git リモート URL を frontmatter に記録する（マシン間でのプロジェクト同定用）
- セッション全体を振り返ってからまとめること
- PR 番号・Issue 番号など外部の可変識別子を文字列で記録する場合は、**ブランチ名・タイトル・最終確認日** も併記する。番号だけだと次セッションで中身を誤認しやすく、実態と食い違った状態でコンテキストが復帰されるリスクがある
