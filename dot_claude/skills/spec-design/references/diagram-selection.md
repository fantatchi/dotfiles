# 図種の判断軸（詳細）

SKILL.md「図種の判断軸（サマリ）」の補足。各図種の典型例・採用基準・テキストソース化ツールを詳しく扱う。

## C4 model（システム俯瞰）

4 階層: Context / Container / Component / Code。

- **Context**: システムを 1 つの箱として外部システム・ユーザーとの関係を示す。最も粒度が粗く、非エンジニアにも見せられる
- **Container**: アプリケーション・データストア・ファイルシステムなど、デプロイ単位の構成
- **Component**: Container 内部の論理コンポーネント。クラス図より粗い
- **Code**: クラス図相当。通常は描かない（IDE で十分）

**実務では Context + Container の 2 階層で 80% 足りる**。Component は本当に込み入った Container だけ補足する。

## UML の取捨選択

### 採用（3 種）

| 図 | 使いどころ | 注意 |
|---|---|---|
| **クラス図（軽量）** | データモデル / ドメインモデル | 全属性・全メソッドを書かず、関心領域だけ |
| **シーケンス図** | API/サービス間の時系列呼び出し | 1 呼び出しなら表で足る |
| **状態遷移図** | エンティティのライフサイクル | 状態 3 つ以下なら表で足る、From×To マトリクスも有効 |

### 不採用または最小限

- **ユースケース図**: 棒人間と楕円の関係性を示すだけで情報量が乏しい。**ユースケース「記述」（テキスト）の方が役立つ**
- **コンポーネント図**: C4 Container でカバー可能
- **配置図**: C4 Container でカバー可能（物理ホスト・ネットワーク図が本当に必要なら別途）
- **オブジェクト図 / パッケージ図 / アクティビティ図 / コミュニケーション図 / タイミング図 / 複合構造図 など**: 仕様書としての ROI が低い

### 結論

**C4 で骨組みを描き、必要箇所だけ UML を差し込む** のが現実解。「UML 全種類を網羅すべき」発想は捨てる。

## BPMN（業務プロセス）

人・組織・システムが混在する業務プロセス・申請フロー・承認フローを描く。

- **プール / レーン** で関係者を分離（受注者 / 発注者 / システム）
- 簡易フローで足りる場合（1 アクター完結など）は箇条書き手順書を選ぶ
- 例外フロー（差し戻し・キャンセル）も同じ図に乗せると遷移漏れが見つかりやすい

draw.io / Bizagi Modeler が無償で扱える。

## ツール選択（テキストソース化）

PNG 直貼りは更新されなくなるアンチパターン。**テキストでバージョン管理できる形式** を選ぶ:

| 形式 | 強み | 弱み |
|---|---|---|
| **Mermaid** | GitHub / Obsidian 標準サポート、軽量、Markdown 内に埋め込める | レイアウト自動配置のため細かい制御は難しい |
| **PlantUML** | 表現力豊富（UML 全種類対応） | 描画にサーバまたは jar 実行が必要 |
| **draw.io（XML）** | GUI で複雑な図も描ける、XML がテキスト diff 可 | バージョン管理向きとはいえ XML diff は読みにくい |
| **SVG（手書き or ツール出力）** | フォント・配色を完全制御可、CSS で再スタイル可 | 大きい図は手書き辛い |

**HTML 主体プロジェクトでは SVG 直書きが意外と扱いやすい**（CSS 変数で配色を一元化できるため、ドキュメント全体とテーマを揃えられる）。Mermaid 標準で足りるならそれが最短。

## 図のキャプションルール

すべての図に「**この図で何が言いたいか**」を 1 行で添える。例:

- 悪い例: 「図 3-1: システム構成」
- 良い例: 「図 3-1: 認可は API Gateway 層で完結し、バックエンドサービスは認可済み前提で動作する」

キャプションを書けない図は「何を伝えたいか曖昧」のサインなので削るか描き直す。

## 状態遷移を「表」で書く選択肢

状態が 4-8 個程度なら、状態遷移図より From×To マトリクスのほうが読みやすい:

| From / To | Draft | Submitted | Approved | Rejected |
|---|---|---|---|---|
| Draft | — | submit | — | — |
| Submitted | recall | — | approve | reject |
| Approved | — | — | — | — |
| Rejected | revise | — | — | — |

セル = 遷移を起こすアクション、空白 = 不可。図より「不可遷移の網羅性チェック」が機械的に行える。
