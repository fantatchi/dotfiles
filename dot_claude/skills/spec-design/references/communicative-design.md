# 伝わるデザイン原則（配色以外）

仕様書・設計ドキュメント（HTML / PDF / md）の視覚デザインを「伝わりやすく」整えるための原則集。SKILL.md「## 出力フォーマットのすみ分け」「### Step 3: スタイルを既存に合わせる」から呼び出される。

**出典**: 「伝わるデザイン」(<https://tsutawarudesign.com/>) — 各原則の末尾に該当ページ URL を明記。

**範囲外**: 配色（カラーパレット選定・コントラスト比・色覚多様性配慮）は本ファイルでは扱わない。配色は [`../../shared/base-color-mapping.md`](../../shared/base-color-mapping.md) および [`../../dashboard-design/references/visual-encoding.md`](../../dashboard-design/references/visual-encoding.md) を参照。

## 適用範囲

- 新規 HTML 補足ページ生成時の CSS 設計
- PDF / スライドで仕様書を配布する時のレイアウト判断
- md ファイルでも適用可能な原則（行頭揃え、箇条書きの統一など）

既存スタイルがある場合は SKILL.md の Step 1 に従い**既存踏襲を優先**。本ファイルの原則は新規プロジェクトおよびリファクタ判断の基準として使う。

---

## 原則間の優先順位マトリクス

12 原則は独立ではなく、適用時に競合・優先関係がある。矛盾を感じたら以下に従う:

| 関係 | 優先する原則 | 劣後する原則 | 判断指針 |
|---|---|---|---|
| 整列 vs 視覚バランス | **整列（原則 1）** | センタリング装飾 | 左揃え統一を死守 |
| 近接 vs 等間隔余白 | **近接（原則 2）** | 余白の均等化（原則 6） | 関係性が見える方を優先 |
| ジャンプ率 vs 視線の流れ | 状況依存 | — | 主要要素は左上、強調は右下に逆配置可 |
| 強調手段 | **太字（原則 9）** | 下線・斜体・枠囲み | 太字 1 種類で足りる範囲に絞る |
| 反復 vs 多様性 | **反復（原則 3）** | 章ごとの個別装飾 | 同種要素は同じ見た目で統一 |

判断つかない場合は **「読者が詰まる場所はどこか」** を優先軸にする。

## 構造の原則（1〜5）

### 1. 整列を最優先する

- **やる**: 左揃え・上揃えを基本とし、見出し・本文・表セルの行頭をすべて同じ位置に揃える
- **やらない**: センタリングを多用したり、要素ごとに揃え方を混在させない
- **仕様書への適用例**: HTML/CSS で `text-align: left` を既定とし、コードブロック・図キャプション・表セルの開始位置を全 1 カラム上で揃える。md でも見出し下の本文インデントは入れない
- **出典**: <https://tsutawarudesign.com/miyasuku4.html>

### 2. 近接でグループを示す

- **やる**: 関連する要素（見出しと本文、図とキャプション、項目と注記）を物理的に近接させ、無関係な要素とは距離を取る
- **やらない**: 等間隔の余白で全要素を並べ、関係性を視覚的に潰さない
- **仕様書への適用例**: HTML 補足ページで section 間 `margin-top` を section 内の段落間隔より明確に大きく取る。PDF では見出し直後の余白を本文段落間より詰める
- **出典**: <https://tsutawarudesign.com/miyasuku4.html>

### 3. 反復で構造の予測可能性を上げる

- **やる**: 同種要素（注意ブロック・コード例・図キャプション）には同じ書式パターンを繰り返し適用する
- **やらない**: 同じ意味の要素を章ごとに違う見た目で表現しない
- **仕様書への適用例**: ADR の各 entry を必ず同じ見出し階層・同じプロパティ順で記述。HTML では `.note`, `.warn`, `.example` といったクラスを定義し全章で使い回す
- **CSS 集約による視覚一貫性の担保（HTML 補足ページ複数時は SHOULD）**: 装飾を共通 CSS（例: `_shared/spec-page.css`）に集約し、各 HTML は `:root` 固有変数のみとする。1 ファイル変更で全ページに伝播。詳細手順は [`html-css-centralization.md`](./html-css-centralization.md) 参照
- **出典**: <https://tsutawarudesign.com/miyasuku4.html>

### 4. ジャンプ率・コントラスト・視覚的階層 3 段まで

- **やる**: 見出しと本文のサイズ比を明確に大きく取り、階層が一目で分かるようにする。サイズ・太さの対比（コントラスト）で重要度を表現する。**視覚的階層は 3 段までに抑える**（H1 / H2 / H3 のみを「見出し階層」として扱い、H4 以下は本文と同サイズの太字で代替）
- **やらない**: H1〜H3 を 1〜2pt 差で並べて区別不能にしない。階層を 4 段以上に深めない（読者が「自分が何階層目にいるか」を見失う）
- **仕様書への適用例**: HTML で H1=2.0rem / H2=1.5rem / H3=1.2rem / body=1.0rem 程度の段差を確保。H4 以下は `font-size: 1rem; font-weight: 700`。PDF も同等比率。配色によるコントラストは `shared/base-color-mapping.md` のベースカラー 900 階調を採用
- **出典**: <https://tsutawarudesign.com/miyasuku4.html>（ジャンプ率・コントラストの章）

### 5. 視線の流れ（左上→右下）に沿わせる

- **やる**: タイトル → 概要 → 詳細 → 補足の順で左上から右下へ自然に流れる配置にする
- **やらない**: タイトルより上や左に本文・図を置かない
- **仕様書への適用例**: README の冒頭は「これは何か」を最上部に置き、図解は説明文の右または下に配置。HTML の two-column 構成では主情報を左に置く
- **出典**: <https://tsutawarudesign.com/miyasuku4.html>

## 余白とタイポグラフィの原則（6〜9）

### 6. 余白は要素として設計する

- **やる**: 本文文字 1 文字分以上を最小余白とし、枠と文字、図と周辺要素が触れないようにする
- **やらない**: 紙面・画面を埋めようとして要素同士を密着させない
- **仕様書への適用例**: HTML の `<pre>` `<table>` `<blockquote>` に共通の `padding` を設定。PDF のページ余白は本文の 2〜3 文字分以上を確保
- **出典**: <https://tsutawarudesign.com/miyasuku4.html>

### 7. 本文ゴシック・見出しゴシックを基本に、UD フォントを選ぶ

- **やる**: 画面表示・プレゼン用は游ゴシック / メイリオ / BIZ UDゴシックなど可読性の高いサンセリフを採用。和文と欧文は太さ・スタイルを揃えて組む
- **やらない**: MS ゴシック・MS 明朝、装飾書体、画面表示で細る明朝体本文を使わない。英数字に和文フォントを当てない
- **仕様書への適用例**: HTML の `font-family` は OS 互換性を考慮して以下の順で指定:

  ```css
  font-family:
    "BIZ UDPGothic",     /* Windows 10 1809+（同梱フォント） */
    "Hiragino Sans",      /* macOS */
    "Noto Sans JP",       /* Web font / Linux */
    "Yu Gothic",          /* Windows 旧 fallback */
    system-ui,
    sans-serif;
  ```

  PDF も同系統で統一。Web 配信なら Google Fonts の `Noto Sans JP` を `@import` するのが OS 横断で最も安全
- **出典**: <https://tsutawarudesign.com/yomiyasuku1.html>, <https://tsutawarudesign.com/yomiyasuku2.html>, <https://tsutawarudesign.com/universal2.html>
- **注記**: 原典は「游ゴシック / メイリオ」推奨で UD フォント指定は明示されていない。BIZ UDPGothic（Windows 同梱の UD フォント）/ Noto Sans JP（Google Fonts）の優先指定は **アクセシビリティ実務則** として本ガイドが追加した

### 8. 行間・行長を可読域に収める

- **やる**: 行間は文字サイズの 1.5〜1.75 倍程度（本文と本文の間に「ほぼ一文字分」のすきま）。一行の長さは Web 仕様書では和文 40 文字相当（`max-width: 70ch`）を実務則として採用
- **やらない**: 行間を詰めて黒い塊を作らない。一行を画面横幅いっぱいに伸ばさない
- **仕様書への適用例**: HTML body に `line-height: 1.7; max-width: 70ch` を設定。md → PDF 変換時も同等の設定を行う
- **出典**: <https://tsutawarudesign.com/yomiyasuku4.html>
- **注記**: 原典は **プレゼン前提で「和文 10 文字以下」推奨**。Web 仕様書では一行が短すぎると視線移動コストが増えるため、Web タイポグラフィ一般則の **40 文字相当（`70ch`）を実務則として採用** している。プレゼン用途・スライド出力では原典の 10 文字以下を優先する

### 9. 強調は太字で 1 種類、3 つ以上重ねない

- **やる**: 強調は太字（または太いゴシック）を基本に 1 手段で行う
- **やらない**: 太字 + 下線 + 斜体 + 枠囲みを同時に使わない。明朝体本文に擬似ボールドをかけない
- **仕様書への適用例**: md/HTML では `**bold**` を一次強調、`<mark>` や callout box を二次強調までに留める。注意喚起は装飾の重ねがけでなく callout の種類で区別
- **出典**: <https://tsutawarudesign.com/yomiyasuku4.html>

## 要素別の原則（10〜12）

### 10. 箇条書きは行頭・ぶら下げ・階層を統一する

- **やる**: 左端と 2 行目のぶら下げ位置を統一。階層は記号（●→・→-）と字下げで明示。並列項目は文法構造を揃える
- **やらない**: 空行で項目間隔を稼がない。階層ごとに記号を気まぐれに変えない。文末が「です/だ調」で混在しない
- **仕様書への適用例**: md でリストのネストはスペース数を統一（2 or 4 で揃える）。HTML では `ul ul` に異なる `list-style` を割り当てて視覚階層を明示
- **出典**: <https://tsutawarudesign.com/yomiyasuku5.html>

### 11. 表はデータインク比を高め、罫線を最小化する

- **やる**: 横罫を主、縦罫は最小限。行間を本文より広めに取り、数値は欧文フォントで右揃え。重要列はサイズか太字で強調。**複雑な表（5 行以上 × 4 列以上）は 1 行おきの zebra stripe で視線誘導を補強**
- **やらない**: 全セルに格子罫線を引かない。塗りと枠線の両方に色を付けない
- **仕様書への適用例**: HTML 表に `border-collapse: collapse` + 横罫のみのスタイル。zebra stripe は `tbody tr:nth-child(even) { background: var(--bg-stripe); }` で実現（背景色は Solid Gray 50 `#F2F2F2` またはベースカラー 50 階調）。PDF/md でも数値列を右揃え、単位列は別カラムに分離
- **出典**: <https://tsutawarudesign.com/miyasuku1.html>

### 12. 図解は形状・矢印・サイズを統一し、装飾を削る

- **やる**: 角丸の R 値、ノード矩形のサイズ、矢印の太さを 1 種類に統一。塗りか線かのどちらか一方だけに色を付ける。フローチャートのノードは同サイズ
- **やらない**: SmartArt の影・グラデーション・3D 効果をそのまま使わない。矢印を主役級に太く・派手にしない。楕円と円を混在させない
- **仕様書への適用例**: Mermaid / PlantUML / draw.io 図でスタイルを 1 set に固定（テーマファイルを共有）。HTML 補足ページのインライン SVG にも同じスタイル変数を適用
- **出典**: <https://tsutawarudesign.com/miyasuku3.html>

## 番外: 約物まわりの即効ルール

仕様書では数字・記号の扱いも読みやすさに直結する。本文 12 原則と分離しているのは「組織既存スタイルに従う」余地が大きいため:

- 和文中の括弧は全角（　）、欧文中は半角 ( ) を使い分ける
- 数字と単位は「数字を大きく / 単位を一回り小さく」（HTML なら `<span class="unit">`、PDF/md でも視覚比を確保）
- 数値は欧文フォント、和文中に欧文を混植する時は太さ・サイズを和文に揃える
- 文末の `?` `!` は半角、句読点は `、` `。` で統一（混在を避ける）

**組織スタイル例外**: 学術系（理系論文 / 工学系文書）は伝統的に `，．` を使う。組織既存ドキュメントが `，．` の場合はそれを踏襲し、SKILL.md Step 1 で確認する。

**出典**: <https://tsutawarudesign.com/yomiyasuku6.html>

## レビューチェックリスト（伝わるデザイン観点）

新規 HTML / PDF 出力時に確認:

**構造（1〜5）**
- [ ] 整列が左揃え・上揃えに統一されているか（原則 1）
- [ ] 関連要素が近接・無関係要素が離されているか（原則 2）
- [ ] 同種要素のスタイルが反復されているか（原則 3）
- [ ] 見出しと本文のジャンプ率が十分か / 視覚的階層が 3 段以内か（原則 4）
- [ ] 視線の流れが左上→右下に沿っているか（原則 5）

**タイポグラフィ（6〜9）**
- [ ] 余白が要素として設計されているか（原則 6）
- [ ] フォントは UD 系・サンセリフ統一、OS 互換 fallback ありか（原則 7）
- [ ] 行間 1.5〜1.75 倍、行長 40 文字相当 / 70ch か（Web 仕様書時、原則 8）
- [ ] 強調は太字 1 種類に絞られているか（原則 9）

**要素別（10〜12）**
- [ ] 箇条書きの記号・字下げ・文法構造が統一されているか（原則 10）
- [ ] 表は横罫主体・数値右揃え・複雑表は zebra stripe か（原則 11）
- [ ] 図解はスタイル 1 set・装飾削除されているか（原則 12）

**約物**
- [ ] 括弧の全角/半角が使い分けられているか
- [ ] 句読点が組織スタイル（`、` `。` / `，．`）で統一されているか
