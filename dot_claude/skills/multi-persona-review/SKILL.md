# Multi-Persona Review

独立した複数の専門観点を並列で走らせて、単独レビューでは見落とす死角を潰すスキル。

## このスキルが解く問題

1 人のレビュアー（メインエージェント）は **1 つの観点に固着しがち**。特に以下のバイアスを生む:

- **可用バイアス**: 直前に読んだ知識・経験で物語を組み立て、それで仮説確定
- **整合性バイアス**: ある仮説で全観測が説明できると、他の仮説が同じく説明できることに気づきにくい
- **専門領域偏り**: バックエンド側の人間が UI/UX や運用観点を軽視するなど

並列で違う専門観点を走らせると、各エージェントが異なる前提で同じ素材を読むため、**整合的物語ではなく区別可能性で仮説評価**できる。

## 使うべき時 / 使うべきでない時

**使う**:
- インシデント対応の post-mortem で死因仮説を評価したい
- 重要な設計判断（アーキテクチャ選択 / migration 戦略）の見落とし潰し
- runbook / 仕様書を本番投入前に第三者目線で監査
- 自分が立てた仮説を「整合的物語に飛びついていないか」検証したい
- 3 つ以上の独立した専門観点に分解できる

**使わない**:
- 単純なバグ修正・1 ファイル編集
- ユーザーが「軽く見て」「急いでる」と言っている
- 並列化の余地が小さい（1 観点で十分なタスク）
- 破壊的操作を伴う作業（読取専用前提のスキル）

## 実行手順

### Step 1: ペルソナ設計（3〜5 人）

レビュー対象を分析し、**観点が独立な専門ペルソナ**を 3〜5 人選ぶ。

設計の指針:

- **専門領域が重ならない**: 「バックエンド」「フロントエンド」「インフラ」のような表層分類より、「Python 並行性」「Windows プラットフォーム」「SRE 故障診断」のように **解くべき問題に効く専門性** で切る
- **3 人を基本、複雑なら 5 人まで**: 2 人だと観点不足、6 人以上だと統合コストが見合わない

ペルソナの典型例:

| 領域 | ペルソナ例 |
|---|---|
| 障害分析 | Windows プラットフォーム / Python 並行性 / SRE 故障診断 |
| 設計レビュー | アーキテクト / セキュリティ / ペネトレ / SRE |
| コードレビュー | シニア開発 / セキュリティ / テスト設計 / アクセシビリティ |
| runbook 監査 | 運用エンジニア / インシデント対応 / 新人視点 |

### Step 2: 共通プロンプトのテンプレート設計

各ペルソナに渡すプロンプトに **必ず含める要素**:

1. **ペルソナ定義**: 「あなたは XX エンジニア (XX に詳しい人) です」
2. **状況概要 (確認済の事実)**: 何が起きていて、何が確定済みか。**仮説や推測は分けて書く**（先入観を植え付けないため）
3. **前提の参照ポイント**: 読むべきリポジトリ / ファイル / コミット / ログ
4. **レビューしてほしい観点 (4〜6 個の番号付き質問)**: ペルソナ固有の専門知識から答えてほしいこと
5. **アウトプット形式**: 字数制限 (500〜800 字推奨) / 「最も効きそうな対策トップ 3」を箇条書きで明示
6. **読み取り専用の指示**: コード変更・設定変更しないこと、Read / Glob / Grep / WebSearch を必要に応じて使ってよい

**重要な書き方のコツ**:

- ❌「真因を特定して」→ 結論の押し付けに見える
- ✅「残仮説 ① ② ③ のうち、あなたの専門知識から見て蓋然性順位はどうか? それぞれの追加検証手段を 1〜2 個ずつ」→ 専門知識を引き出す質問

- ❌「コードを直して」→ スキルの読取専用前提に反する
- ✅「最も貢献度が高そうな対策トップ 3 を箇条書きで」→ 提案を集めるだけ

### Step 3: 並列起動

`Agent` ツールを **同一の assistant メッセージ内** に N 回呼ぶ。Claude Code は同一メッセージ内の独立したツール呼出を並列実行するため、N 人分のレビューが同時に走る（直列の 1/N 時間で完了）。

要点:

- **subagent_type**: 基本 `general-purpose`（read/write 全権付きだが、プロンプトで読取専用に縛る）。コード探索が中心なら `Explore` も選択肢
- **依存関係**: ペルソナ間に依存がない（A の結果を B が見る必要がない）こと。これは並列化の前提。横断的な議論はメインエージェント側で Step 4 で行う
- **prompt の自己完結性**: 各 prompt は他のペルソナのことに言及しない（各エージェントは互いを知らない）。「他のレビュアーと違う観点で」のような相対的な指示は不可
- **description**: Agent 呼出の `description` には「<ペルソナ名 short> review of <対象 short>」のような短い識別子を入れる（並列実行のログで判別しやすい）

### Step 4: 統合判断（メインエージェント）

3〜5 人の所見が返ってきたら、メインエージェントが統合する。やること:

1. **重要な新発見を抽出**: 各ペルソナが独立に挙げた指摘のうち、メイン側で見落としていた論点を 3〜5 つピックアップ。表組みで提示
2. **仮説の再評価**: 各ペルソナが提示した残仮説を統合し、蓋然性を整理。**唯一統一的に説明できる仮説**があれば最有力として明示
3. **推奨アクション**: 各ペルソナの「対策トップ 3」を統合し、**Tier 1 (即時) / Tier 2 (短期) / Tier 3 (構造変更)** に分類して提示
4. **Tier 3 の判断保留**: 観測なしで構造変更を入れない原則を遵守。「証拠取得後に判断」と明示

### Step 5: ユーザーへの提示

統合所見を以下の構造で提示:

```
## 統合所見 — 重要な新発見 N 個

### 1. <発見タイトル>（誰の指摘）
<内容>

### 2. ...

---

## 推奨実装プラン（優先順）

| 案 | 内容 | 時間 |
|---|---|---|
| α | Tier 1 + Tier 2 | NN 分 |
| β | Tier 1 のみ | NN 分 |
| γ | 全部 (Tier 3 含む) | NN 分 |

推奨: <推奨案> (理由)。どれで進めますか?
```

ユーザーの選択を待ち、承認された範囲のみ実装に進む。

## よくある失敗と回避策

| 失敗 | 回避策 |
|---|---|
| ペルソナの専門領域が重複し似た所見が並ぶ | 「バックエンド」のような表層分類ではなく、解くべき問題に効く細粒度の専門性で切る（例: 「フロントエンド」→「アクセシビリティ」「パフォーマンス」「i18n」） |
| 各ペルソナへの状況説明に「結論」を書いてしまい誘導 | 状況概要は「確定事実」と「仮説」を明確に分けて書く。仮説は「現時点でこう推測している」と明示し、ペルソナに評価を任せる |
| アウトプットが長すぎて統合コストが膨らむ | 各 prompt で字数制限を明示（500〜800 字推奨） + 「最も効きそうな対策トップ 3」を箇条書きで要求 |
| 統合せず所見をそのまま並べて終わる | Step 4 の統合は必須。新発見抽出 + 仮説再評価 + Tier 分類で初めて価値が出る |
| Tier 3 の構造変更を観測なしで投入してしまう | 「証拠取得後に判断」を原則として Tier 3 は保留、ユーザー判断を待つ |
| 統合判断で「もう 1 観点欲しかった」と感じる | 追加で 1 人だけ起動するのも可（コスト増を許容するなら）。ただし最初のペルソナ設計時に専門領域カバレッジを意識するのが本筋 |
