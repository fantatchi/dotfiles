# ワークフロー: 仕様書の新規作成・既存改善・HTML 補足

## 新規でドキュメントを作る場合

1. **読み手と目的を確認** — 誰のため・何のため・規模・既存状態の 4 点（SKILL.md の Step 1 参照）
2. **推奨構成から必要な部分を選定して提案** — [docs-tree.md](./docs-tree.md) の最小パッケージから始める
3. **着手順で骨格を作る** — README → 用語集 → C4 Level 1 → 最初の ADR
4. **機能仕様は実装と並行して書く** — 事前に書きすぎない
5. **リポジトリの `docs/` に配置する md として生成** — Docs as Code で運用

## 既存ドキュメントを改善する場合

1. **「3 つの詰まりポイント」診断** — 既存ドキュメントを読み、以下がカバーされているかチェック:
   - 全体像: README / C4 Level 1 が存在し、5 分で全体像が掴めるか
   - なぜ: 過去の意思決定の背景が ADR に残っているか
   - 用語: 用語集が「唯一の出典」になっており、本文で再定義されていないか
2. **欠けている部分から優先的に追加** — 3 つの詰まりポイントを最優先
3. **既存の冗長な詳細は削減を提案** — 一気に消さず、ユーザー確認しながら段階的に
4. **古い図・本文不整合の解消** — レビュー時に「図のキャプション」を必ず読み合わせる

## HTML 補足ページを作る場合

md がメインの運用で、視覚情報が主役のページを HTML 補足する場合のワークフロー:

### HTML 補足を選ぶ判断基準

以下のいずれかに該当する場合、HTML 補足が向いている:

- **サマリー / 概況ページ** — KPI 一覧、進捗一覧、ステータス可視化
- **システム概要ランディング** — 「ぱっと見で構造を伝えたい」入口ページ
- **比較・対比ページ** — 選択肢の比較表、配色で意味を伝える表
- **配色で情報を符号化したい表** — 状態（OK/警告/NG）を色 + アイコンで表現

逆に、以下は **md で十分** なので HTML 化しない:

- 仕様本文・API リスト・ADR・用語集 → md
- 簡易図（クラス図・シーケンス図・フローチャート）→ md + Mermaid / PlantUML
- 詳細な記述・コードブロック → md

### HTML 補足ページ作成手順

1. **視覚設計は [visual-encoding.md](./visual-encoding.md) を参照** — 配色 / コントラスト比 / 色数制約 / 装飾排除 / アクセシビリティ / 図表タイトル命名の判断はそちら
2. **既存 HTML 補足のスタイル踏襲** — プロジェクトに `_html/` 配下の既存ファイルがあれば CSS / `:root` 変数をコピー
3. **最小骨格は [templates.md](./templates.md)** — サマリーページ / 比較ページのテンプレを起点に
4. **配置は `docs/_html/` 配下に集約** — md 中心の運用と境界を明確に（詳細は [docs-tree.md](./docs-tree.md)）

### 手書き HTML の運用警告（重要）

手書き HTML 補足ページは **半年で腐る** リスクが高い。以下を必ず守る:

- **本当に視覚情報が主役のページに限定** — 「配色で意味を伝える表」「Z 型レイアウトで構造を一望する概況ページ」「ダッシュボード性のあるサマリー」など。**「md でも表現できるが見栄えがいいから HTML」は禁物**。情報密度・grep しやすさ・PR レビュー差分の読みやすさが md より明確に劣る
- **ビルド生成を優先** — md + Mermaid/PlantUML + MkDocs / Docusaurus / Astro でビルド生成できる場合はそちらを採用。手書き HTML は最後の手段
- **本文 md との同期問題** — 仕様変更のたびに HTML 側も手で更新する必要がある。SoT を md 側に明確化し、HTML 補足は「読み取り専用ビュー」と位置付ける
- **配置を `_html/` 配下に集約** — `docs/*.md` と混在させない。CI ビルド成果物との境界も明確にする
- **腐敗チェック** — 四半期に一度程度、HTML 補足ページの内容と md 本文の整合性を確認する仕組み（PR テンプレ・チェックリスト）を用意

## 出力形式の運用ルール

- デフォルトは **Markdown** で生成
- ファイル名は kebab-case（例: `0001-use-postgresql.md`）
- ADR は連番管理（`0001`, `0002`, ...）
- 図は PlantUML または Mermaid のコードブロックで md に埋め込み
- HTML 補足は `docs/_html/` 配下に直接 HTML として配置（または md からビルド）

## 注意事項

- **書きすぎない** — 維持コストが品質を殺す。読まれない仕様書は害悪
- **コードと同期させる仕組みを必ず提案** — 古い仕様書は嘘より悪い
- **「生きた文書」として扱う** — 一度書いて終わりではない
- **読者の目線を忘れない** — 自分が当然と思うことこそ書く価値がある（特に「全体像 / なぜ / 用語」の 3 点）
