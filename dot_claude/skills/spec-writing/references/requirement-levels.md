# 要件レベル語（詳細）

SKILL.md「要件レベル語（サマリ）」の補足。RFC 2119 / RFC 8174 ベース。

## レベル別の使い分け

### MUST（必須）

- 違反は **仕様違反**
- 実装が満たさない場合、その実装は仕様に準拠していないと判断される
- 同義語: SHALL、REQUIRED
- 日本語: 「しなければならない」「必須である」

例:
- 認証トークンの有効期限は **MUST** 24 時間以内であること
- ファイルアップロードサイズは **MUST** 100 MB 以下に制限すること

### MUST NOT（禁止）

- 違反は仕様違反
- 同義語: SHALL NOT
- 日本語: 「してはならない」

例:
- パスワードは平文で保存 **MUST NOT**
- API レスポンスにスタックトレースを含め **MUST NOT**

### SHOULD（推奨）

- 違反する場合は **理由を残す**（コメント / ADR / コミットメッセージ）
- 例外的な実装でも仕様違反とはみなされない
- 同義語: RECOMMENDED
- 日本語: 「推奨する」「望ましい」

例:
- レスポンスタイムは **SHOULD** 200 ms 以下であること（推奨）
- エラーログには correlation_id を **SHOULD** 含めること

### SHOULD NOT（非推奨）

- 同義語: NOT RECOMMENDED
- 日本語: 「推奨しない」「避けるべき」

### MAY（任意）

- 実装するかどうかは自由
- 同義語: OPTIONAL
- 日本語: 「してよい」「実装してもよい」

例:
- 管理者は監査ログを CSV エクスポート **MAY** できる
- ダークモードを **MAY** サポートしてよい

## アンチパターン: 曖昧な表現

以下は要件レベル語に置き換える:

| 曖昧表現 | 推奨置換 |
|---|---|
| 「〜する」 | MUST または SHOULD のどちらか明示 |
| 「〜できる」 | MAY |
| 「基本的に〜」 | SHOULD（例外を許容する意図） |
| 「原則として〜」 | SHOULD |
| 「可能な限り〜」 | SHOULD（具体閾値があれば MUST に格上げ） |
| 「適宜〜」 | 削除（実装者に判断を丸投げしている） |
| 「望ましい」 | SHOULD |
| 「必要に応じて」 | MAY または SHOULD（条件を明示） |

## QA / AC 抽出への効果

要件レベル語が統一されていると、QA は以下を機械的に抽出できる:

- **MUST/MUST NOT の文 → AC（Acceptance Criteria）の候補**
- **SHOULD の文 → 例外パス・エッジケースの確認対象**
- **MAY の文 → スコープ判断対象（実装しない選択もアリ）**

逆に曖昧表現が残っていると、QA が「これはテストすべきか？」を判断するために毎回ステークホルダーに確認する必要が出る。

## 日本語仕様書での運用

日本語仕様書では、英語の助動詞をそのまま埋めるか、対訳の日本語を選ぶか、プロジェクトごとに決める。例:

- **A 案（英語埋め込み）**: 「認証トークンは MUST 24 時間以内であること」
- **B 案（日本語対訳）**: 「認証トークンは 24 時間以内で **なければならない**」
- **C 案（太字統一）**: 「認証トークンは **必須** で 24 時間以内」

どれでも良いがプロジェクト内で揃える。**揃っていることが大事**（grep で AC 候補を機械抽出できる）。
