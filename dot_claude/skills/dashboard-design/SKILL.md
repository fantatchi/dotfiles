# Dashboard Design

ダッシュボード設計の専門家として、デジタル庁「ダッシュボードデザインの実践ガイドブック」（2026 年 3 月更新版）に基づきチャート選択・配色・レイアウト・命名を導く。

原典: https://www.digital.go.jp/resources/dashboard-guidebook

## 本スキルの対象範囲

本スキルは **提示型ダッシュボード**（概況の把握 / 一般的な知識 / 単純な操作）を主対象とする。経営ダッシュボード、KPI モニタリング画面、業務報告画面、現場進捗ダッシュボードなどが該当する。

**探索型ダッシュボード**（特定ドメイン知識前提 + 複雑な操作で詳細分析を行うもの。BI 分析者向けインタラクティブツールなど）は対象外。

## 使うべき時

- ダッシュボード / KPI 画面 / レポート画面の構成相談
- 既存ダッシュボードのレビュー（読み取りやすさ・配色監査）
- グラフ種の選択（折れ線か棒か円か面か）
- ダッシュボードのタイトル・凡例・注釈の文言設計
- データ可視化のアクセシビリティ確認

## 使うべきでない時

- 単発のチャート実装（ライブラリ選定、コーディング） → `frontend-design`
- ブランド配色の適用 → `theme-factory` / `brand-guidelines`
- データパイプライン・SQL 設計
- インフォグラフィック・印刷物デザイン → `canvas-design`
- 探索型ダッシュボード（複雑な操作・特定ドメイン知識が前提のもの）。本スキルは**提示型ダッシュボード**（概況把握・一般知識・単純操作）を主対象とする

## 設計の 5 原則

1. **データに合ったグラフ種を選ぶ** — 時間変化→折れ線、量比較→棒（円より優先）、構成比+時間→積み上げ面。詳細は `references/chart-selection.md`
2. **装飾より情報密度** — 3D / ドロップシャドウ / 過剰グリッドを排除。色数は 1〜5 色に絞る。詳細は `references/visual-encoding.md`
3. **視線フローに沿った階層配置** — 左上に全体指標、右下に詳細。フィルターは上部または左部。詳細は `references/layout-and-hierarchy.md`
4. **タイトル・凡例で文脈を補う** — タイトルは「対象 + 指標 + データ種別」を明記、凡例は隣接配置・順序対応。詳細は `references/titles-and-context.md`
5. **アクセシビリティ** — コントラスト比 3:1 以上、色のみで分類しない、データテーブル/代替テキスト/要約テキストを併用

## 設計フロー（4 ステップ）

### Step 1: 要件確認
- 誰が見るのか / 何を判断する画面か
- 提示型か探索型か（本スキルは提示型のみ対象 — 探索型は対象外）
- データソース・更新頻度
- 制約条件（画面サイズ、利用環境、ブランディング要件）
- 関係者からの要望ヒアリング時は **要望そのものではなく「理由・意図」を聞き出す**（「○○したいから、こうしたい」の○○を引き出す）

> 詳細な要件整理プロセスは原典 PDF 第 2〜3 章を参照

### Step 2: 情報の優先度と階層化
- 全体指標（最初に見るべき大きな指標）を 1〜3 個特定
- 詳細指標（判断・行動に役立つ細かい指標）を整理
- 全体 → 詳細の階層構造で配置設計

> **詳細は `references/layout-and-hierarchy.md` を参照**（視線フロー、グリッドシステム、フィルター配置、レイアウトサンプル型）

### Step 3: 各情報のグラフ種選択
- データの軸（時間 / カテゴリ / 連続値）を特定
- 伝えたい情報（時系列推移 / 比較 / 構成比）から最適なグラフ種を選ぶ
- 円グラフは全体総量が明らかでコンパクト表示が必要な場合のみ、それ以外は棒へ

> **詳細は `references/chart-selection.md` を参照**（グラフ種別マトリクス、各グラフの使用条件、選定の判断軸）

### Step 4: レイアウト・配色・タイトル決定
- グリッド（1920×1080 16:9 / 縦横 6 分割）に沿って配置
- カラーパレット（7 系統 × 6 階調）から色覚多様性配慮の配色を選定
- タイトルは「国産自動車の出荷台数（月次推移）」型で命名
- 異常値・遅延・警告は色 + 形/アイコンで二重符号化

> **配色 (HEX 含む) は `references/visual-encoding.md`、タイトル・凡例・比較対象併記は `references/titles-and-context.md` を参照**

## レビューチェックリスト

ダッシュボードを作成・レビューするときに最低限確認する項目（PDF 第 5.4 節を圧縮）:

### 利用者の体験
- [ ] 利用者の判断・行動に必要な情報が提供できているか
- [ ] 全体像を捉えるための基準指標が掲載されているか
- [ ] 本当に必要な情報だけに絞り込めているか
- [ ] 数回の操作で目的が達成できるか

### 指標・表・グラフのデザイン
- [ ] **棒グラフの原点は 0 か**
- [ ] グラフの色数は **1〜5 色** に絞れているか
- [ ] グラフと凡例が**隣接**し、対応づけが明確か
- [ ] 凡例の順序がグラフの系列順序と**一致**しているか
- [ ] タイトルが「**対象 + 指標 + データ種別**」を含むか（「売上」ではなく「月次売上推移」）
- [ ] 不要な装飾（3D / ドロップシャドウ / 過剰グリッド）を排除しているか
- [ ] グラフ項目の並び順に意味があるか（大小順・更新日順など、意味なきあいうえお順を避ける）
- [ ] 全体指標と詳細グラフが分離配置されているか
- [ ] 比較対象（前年比・目標値・参照値）を併記したか

### データ設計
- [ ] データ定義（対象、意味、単位、更新日）を参照可能か
- [ ] データソース・注釈・免責事項を記載したか

### アクセシビリティ
- [ ] コントラスト比 **3:1 以上** か（無理なら数値近接表示）
- [ ] 色だけで分類していないか（形・塗り模様・ラベルを併用）
- [ ] グラフに代替テキスト（alt）を付与したか
- [ ] データファイル（Excel/CSV）を公開したか（一般公開の場合）
- [ ] 要約テキストを公開したか（一般公開の場合）

## レビュー時の典型的な指摘パターン

| ありがちな問題 | 修正方針 |
|---|---|
| 構成比を円グラフで表示 | 棒グラフに変更（全体総量が明らかでコンパクト表示が必須な場合のみ円） |
| 棒グラフの原点が 0 でない | 原点 0 にする（数量が誇張・矮小化されて見える） |
| タイトルが「売上」だけ | 「月次売上推移（税抜・速報値）」のように対象 + 種別を追加 |
| 色数が多すぎる | 1〜5 色に絞り、注目系列を強調色・他をニュートラルに |
| 凡例がグラフから離れている | グラフに隣接配置、順序も系列と一致 |
| 詳細グラフだけ並んでいる | 左上に全体指標を追加 |
| 並び順があいうえお順 | 数量大小順・更新日順など意味のある順に |
| 3D / ドロップシャドウ多用 | 装飾を排除、情報量を最低限に |

## 他スキルとの連携

本スキルは**設計判断**に特化する。以下のフェーズに進んだら他スキルへ引き継ぐ:

| フェーズ | 引き継ぎ先 | 例 |
|---|---|---|
| React/HTML/CSS でのチャート実装 | `frontend-design` | 「Recharts でこの折れ線を実装したい」 |
| ライブラリ選定 (Recharts / Plotly / D3) | `frontend-design` | 「どの可視化ライブラリが向いている？」 |
| ブランド配色の機械的適用 | `theme-factory` | 「コーポレートカラーを反映したい」 |
| 印刷物・ポスター | `canvas-design` | 「PDF 報告書のデザイン」 |
| PDF レポート出力 | `pdf` | 「ダッシュボードを PDF にしたい」 |

典型フロー: dashboard-design で **何をどう見せるか**を決める → frontend-design で **どう実装するか**を決める。
